2. Security: Supabase Anon Key & Row Level Security

2.1 What is the Supabase Anon Key?

The Supabase anon key is a publishable API key that enables frontend applications to communicate directly with Supabase services (database, auth, storage) through the supabase-js client. It authenticates the application with an anonymous/public role, while user-specific permissions are determined by the user's session JWT after login. This key is safe to include in frontend code when Row Level Security (RLS) is properly configured.

2.3 Why Use the Anon Key Instead of a Custom API?

  • Simplicity & Speed: Eliminates the need for maintaining a custom Node/Express API
  • Built-in Security: RLS enforces access rules at the database level
  • Real-time Capabilities: Native support for database subscriptions
  • Reduced Complexity: Fewer servers to deploy and maintain
  • Type Safety: Generated types help prevent development errors

When NOT to use only the anon key: For operations requiring secret credentials (Stripe secrets, third-party API keys) or full-admin cross-tenant tasks, use Supabase Edge Functions with service role key on the server side.

2.4 Key Security Concepts

Anon Key vs. Service Role Key

Anon Key (Public):

  • Safe for frontend use
  • Limited to "anon" role permissions
  • Cannot override RLS

Service Role Key (Private):

  • Never use in client code
  • Bypasses RLS with full database access
  • Reserved for server-side operations only

2.5 Authentication Flow with RLS

  1. App initializes supabase-js client with anon key
  2. User authentication provides session JWT (stored in localStorage via src/hooks/useAuth.ts)
  3. Subsequent requests include user JWT with user ID (auth.uid())
  4. RLS policies validate permissions per operation
  5. Database enforces row-level access rules

2.6 Row Level Security Implementation

RLS policies ensure that users can only access data they're authorized to view or modify. Key policies include:

User Profiles

  • Users can only access their own profile records
  • Implemented via auth.uid() = id checks
  • Prevents users from viewing or editing other profiles

User Progress

  • Users can only view and modify their own progress data
  • Enforced through user_id = auth.uid() matching
  • Blocks access to anyone else's progress

Tutorials & Content

  • Read access to published content for authenticated users
  • Write operations restricted to admin users only
  • Role-based permissions through profile checks
  • Ensures only admins can create/edit/delete tutorials

Storage Security

  • Public read access for published content
  • Restricted read for enrolled/entitled users
  • Write/delete operations limited to admins or content owners
  • Object-level permission validation

Admin Privileges Implementation

Administrative access is granted through:

  • Role designation in user profiles (user_profiles.role = 'admin')
  • JWT-based role verification through RLS policies
  • Admin actions use the same anon key but elevated privileges through JWT
  • No RLS bypass from client applications

2.7 Security Advantages Over Custom API

  • Database-Level Enforcement: Security rules are applied at the data layer itself
  • Reduced Attack Surface: Eliminates common API endpoint authorization flaws
  • Consistent Policies: Uniform security rules across all access patterns
  • Network Call Protection: Even reverse-engineered requests cannot bypass RLS policies

2.8 Practical Security Implications

  • Progress Tracking: Users can only upsert/select their own user_progress rows
  • Content Management: Only admin JWTs pass RLS checks for updates/deletes
  • File Access: Storage policies ensure only authorized roles can manage files
  • Personalization: Users can only access their own preferences data

2.9 When to Use Server-Side Solutions

Use Supabase Edge Functions with Supabase Secrets for:

  • Operations requiring secret credentials
  • Cross-tenant administrative tasks
  • Batch processing operations that require RLS bypass
  • Any functionality requiring service role privileges

2.10 Common Security Mistakes to Avoid

  • Never disable RLS on tables containing user data
  • Never include the service_role key in frontend code
  • Don't rely solely on client-side checks for security
  • Implement least-privilege policies (start strict, open carefully)
  • Ensure all writes validate ownership or admin role
  • Configure storage bucket policies explicitly

2.11 Security Checklist

  • [ ] RLS enabled on all user data tables
  • [ ] Service role key excluded from client code
  • [ ] Policies follow principle of least privilege
  • [ ] Storage bucket policies properly configured
  • [ ] Admin privileges validated through database roles
  • [ ] All tables have appropriate ownership checks
  • [ ] Authentication properly implemented via useAuth hook

6.4 Row Level Security (RLS) Policies

Database-level security ensures users can only access their own data:

-- Users can only see their own profile CREATE POLICY "Users can view own profile" ON user_profiles FOR SELECT USING (auth.uid() = id); -- Users can only update their own progress CREATE POLICY "Users can update own progress" ON user_progress FOR ALL USING (auth.uid() = user_id); -- Only admins can manage tutorials CREATE POLICY "Admins can manage tutorials" ON tutorials FOR ALL USING ( EXISTS ( SELECT 1 FROM user_profiles WHERE id = auth.uid() AND user_metadata->>'role' = 'admin' ) ); -- All users can view published tutorials CREATE POLICY "Users can view published tutorials" ON tutorials FOR SELECT USING (is_published = true);

The authentication system provides secure user management with multiple verification layers:

User Registration Process

  2. Supabase Registration: User data sent to Supabase Auth service
  3. Email Verification: Automated verification email sent to user

  1. Credential Validation: Email and password verified against database
  2. JWT Token Generation: Supabase creates signed JWT token
  3. Session Management: Token stored securely in localStorage
  4. Automatic Refresh: Tokens automatically refreshed before expiration
  5. State Synchronization: User state updated across all components

Password Security

  • Bcrypt Hashing: Passwords hashed with salt before storage
  • Minimum Requirements: 8 characters minimum length enforced
  • Strength Validation: Frontend checks for password complexity

Token Security

  • HTTP-Only Storage: Sensitive tokens stored securely
  • Expiration Management: Automatic token refresh before expiration
  • Logout Cleanup: Complete session cleanup on logout

API Security

  • Rate Limiting: Supabase built-in request rate limiting
  • CORS Configuration: Proper cross-origin request handling
  • SQL Injection Protection: Parameterized queries prevent injection
  • XSS Protection: Input sanitization and output encoding
